> **摘要**：LLM 是一台概率性引擎——它会幻觉、会漂移、会把 plausible-sounding 的错误说得像真理。从模型厂商到 Agent 产品，从用 AI 写代码的开发者到管理 AI 辅助团队的组织，每一层都在做同一件事：**用确定性的约束框架包裹非确定性的核心，把「信任 AI 会做对」替换成「不需要信任也能确认对」。** 这篇文章将这种模式命名为 Harness Engineering，展开了一个从 Model API 到组织治理的六层分析框架，并论证：AI 时代开发者最稀缺的能力不是写代码，而是设计一套让 AI 产出的代码可靠、可维护、可审计的质量保障体系。
>
> Harness Engineering = 为给定的非确定性系统，设计一套确定性的约束和校验机制，使得最终产出达到预期的质量阈值，同时尽可能少损失产出速度。

# Harness Engineering（一）: Concept Overview

## 核心定义

**Harness Engineering = 用确定性的约束和校验机制，包裹非确定性的核心（LLM / Agent），以保障产出达到预期的质量阈值，同时尽可能少损失产出速度。**

核心公式：

```
Harness Engineering =
  为给定的非确定性系统 (N)
  设计一套确定性的约束和校验机制 (D)
  使得 N × D 的产出达到预期的质量阈值 (Q)
  同时不损失太多产出速度 (V)
```

关键在于：**把「信任 LLM 会做对」替换成「不需要信任也能确认对」。**

> 用户核心洞察：两种 harness 其实都是为其本身要交付的产品/内容，做好对 LLM 不确定性的 harness，来保障产出的质与量。这是同一个模式在不同层级的递归应用。

---

## 一个经典的工程模式

「用确定性包裹非确定性」在软件工程中反复出现：

| 层级 | 非确定核心 | 确定性框架 |
|------|-----------|-----------|
| 硬件层 | 电子噪声、宇宙射线 | ECC 内存、校验位 |
| 网络层 | 丢包、乱序、延迟 | TCP 重传、序号、超时 |
| 分布式系统 | 节点崩溃、时钟偏移 | Raft/Paxos 共识、幂等重试 |
| 数据库 | 并发冲突、磁盘故障 | MVCC、WAL、Replication |
| Model API (Layer 0) | LLM 幻觉、风格漂移 | Structured Output、Tool Use、Guardrails |
| Agent 产品 (Layer 1) | Agent 误判、多步决策错误 | Workflow DSL、Permission、Isolation |
| 开发者 (Layer 2) | Agent 误判需求、Bug、过度工程 | Git、测试、Plan Mode、约束文件 |
| AI 产物适配 (Layer 3) | AI 代码有不同的 bug 模式 | AI 特征检测、针对性 lint 规则 |
| 组织治理 (Layer 4) | 人的认知偏差、AI 协同混乱 | Code Review、CI/CD、RFC 流程 |

每一层都在做同一件事：**识别下层的不确定因素，用机制而不是信任来消除它。**

### Layer 0 — Model API（模型能力来源）

**问题**：裸的 LLM 能续写 token，但不能安全对话、不能 follow complex instruction、输出格式不可控。

**用户**：调用 API / SDK / 本地模型来 build 应用的开发者。

**确定性框架**：

| 机制 | 解决的问题 |
|------|-----------|
| RLHF / DPO / Constitutional AI | 模型不会 follow instruction、输出有害 |
| Structured Output / JSON mode | 输出格式不可控 |
| Tool Use | 让 LLM 调用外部工具而非幻想 |
| Streaming | 长输出的交互体验 |
| System Prompt | 运行时注入约束 |
| Temperature / Top-p | 控制随机性 vs 确定性 |
| Caching | 降低重复调用成本和差异 |

**交付形态**：

| 形态 | 说明 | 例子 |
|------|------|------|
| Managed Model API | 训练+推理+API 一体化，黑盒调用 | Anthropic API, OpenAI API |
| Inference Service | 只托管推理，模型由第三方训练 | Together AI, Groq, Fireworks |
| Raw Model Weights | 只给权重文件，你自己搭推理 | Llama, Mistral, DeepSeek |
| On-device SDK | 模型跑在本地设备上 | Apple Intelligence, Samsung Galaxy AI |
| Batch API | 异步批量处理，非实时 | OpenAI Batch, Anthropic Message Batches |

> **关键认知**：从交付视角，模型训练 / 推理参数 / API 封装对绝大多数消费者来说是一个合并的黑盒——`POST JSON → 拿 JSON`。只有当你自部署推理时，这些内部分层才成为你需要关心的 harness。

### Layer 1 — Agent 产品（构建在 API 之上，让 API 能力变成自主工具）

**问题**：「能调用 tool 的 API」≠「能自主完成多步任务的 Agent」。

**用户**：用 Agent 完成任务的终端用户（可能是开发者，也可能不是）。

**确定性框架**：

| 机制 | 解决的问题 | 实现方式 |
|------|-----------|---------|
| **Workflow DSL** | LLM 编排 LLM 不可靠 | 确定性 JS 控制流（pipeline/parallel/loop） |
| **Schema 校验** | Agent 输出格式不可控 | StructuredOutput 在 tool-call 层闭环验证 + 自动重试 |
| **Pipeline 优先** | Barrier 浪费快任务空闲时间 | 默认 pipeline()，wall-clock = 最慢单链 |
| **Worktree 隔离** | 并行 Agent 文件冲突 | 按需隔离（~200-500ms 开销），未修改自动清理 |
| **Adversarial Verify** | LLM 倾向于 plausible-but-wrong | 多 skeptic 从不同角度证伪，少数服从多数 |
| **Plan Mode** | Agent 不会主动澄清需求 | 实现前先输出方案等待用户确认 |
| **权限模型** | 操作安全 | Human-in-the-loop，渐进式信任，细粒度预授权 |
| **Memory 系统** | 跨会话知识持久化 | 文件即记忆，frontmatter + wikilink，透明可组合 |
| **缓存调度** | 成本 vs 延迟权衡 | ScheduleWakeup 按 cache TTL 设计唤醒间隔 |

> **关键区分**：Layer 0 和 Layer 1 是不同的产品、不同的用户、不同的不确定性来源。Layer 1 直接消费 Layer 0 的 API。Anthropic 同时做了两层（Anthropic API = Layer 0，Claude Code = Layer 1），但它们是独立的 harness 问题。

### Layer 2 — 开发者 Harness（用 Agent 写代码时的质量保障）

**问题**：「我有一个很能写代码但也经常写错的 AI 助手，我要怎么设计我的开发工作流，让最终代码产出是可靠、可维护、可审计的？」

**关键策略**：

| 机制 | 解决的问题 | 实现方式 |
|------|-----------|---------|
| **Git 作为安全带** | Agent 改坏了回不去 | 原子化 commit，出错即 `reset --hard` |
| **CLAUDE.md 约束** | Agent 自由发挥过度 | 运行时约束文件：技术栈、风格、边界、工作流 |
| **Plan Mode 对齐** | Agent 不会主动澄清需求 | 实现前先输出方案等待确认 |
| **小批量迭代** | 一次性生成不可审查 | 拆分为原子化、可测试、可回滚的工作单元 |
| **测试作为合约** | Agent 不知道什么是对的 | 你写测试（定义正确），Agent 写实现（探索如何做） |
| **Worktree 隔离** | 实验性修改污染主分支 | 独立分支/工作树，验证通过 → merge，失败 → 删除 |
| **多轮迭代** | 一次性 3000 行到处都是 bug | 每轮产生可验证中间产物，checkpoint 推进 |
| **审查负担最小化** | 人类注意力是最稀缺资源 | Agent 跑 lint/format，标注关注点，解释做了什么 |

**完整工作流**：

```
┌─────────────────────────────────────────────────────┐
│  1. CLAUDE.md + 约束文件                              │
│     定义项目约束（技术栈、风格、边界）                   │
│     ↓                                                │
│  2. Plan Mode                                        │
│     Agent 先描述方案，你确认后再动手                     │
│     ↓                                                │
│  3. Git checkpoint                                   │
│     干活前先 commit，出错随时回滚                       │
│     ↓                                                │
│  4. 小批量迭代                                        │
│     Agent 实现 → 你审查 diff → 跑测试 → commit           │
│     ↓                     ↑                          │
│     └── 不过关 → reset ──┘                           │
│     ↓                                                │
│  5. Agent 写测试 → 你审查测试逻辑                      │
│     确保测试验证的是「正确的事」                          │
│     ↓                                                │
│  6. Pre-commit hooks                                 │
│     lint + format + type-check + test                │
│     机器自动挡住低级错误                                │
│     ↓                                                │
│  7. Code Review（人类或另一个 Agent）                  │
│     第二个视角发现第一个视角的盲区                       │
│     ↓                                                │
│  8. CI/CD                                            │
│     完整的自动化验证流水线                              │
└─────────────────────────────────────────────────────┘
```

### Layer 3 — AI 产物特征适配

**问题**：AI 写的代码，bug 分布和人类不同。传统 SRE 假设代码是人类写的，bug 有统计规律；但 AI 产物的失败模式有自己的特征。

**这不是传统的 SRE**——它是对「AI 产出」这个特定输入源的针对性质量把关。

| 人类容易犯的错 | AI 容易犯的错 |
|---------------|-------------|
| off-by-one、空指针、边界遗漏 | 幻觉 API、不存在的依赖 |
| 逻辑错误但语法正确 | 看似正确但逻辑不对 |
| 过度保守 | 过度工程 |
| 漏掉错误处理 | 编造不存在的错误处理 |
| 测试覆盖不足 | 测试 mock 了一切但没测实际逻辑 |

**具体实践**：

```bash
# 传统 lint 不会报，但 AI 常干
grep -r "import.*from.*['\"].*-v\d" src/   # AI 编造的包版本
grep -r "TODO.*implement later" src/        # AI 留下的占位符
# 检查 AI 写的测试是否 mock 了一切但无实质断言
# 检查是否引入了不存在的 API 调用
```

> **为什么这一层成立**：因为 LLM 的不确定性在 dev 阶段已经留下了痕迹——产出的代码有其独特的风险画像。传统工程工具（linter、类型检查、测试）cover 不到这些「AI 特征 bug」，所以需要一层专门的适配。

### 关于「Runtime AI Harness」的讨论和否定

**初始假设**：AI 产品上线后，LLM 在生产中被实时调用，需要一层专门的运维 harness（canary deploy、feature flag、rollback 等）。

**用户的关键质疑**：

> 「上线安全、故障兜底这些在传统软件开发中也是有的。如果在 Layer 2 输出基础上没有新的 AI 使用引入，那就不算 harness engineering。」

**结论**：这一点成立。经过辨析——

- 如果 AI 只在开发阶段使用（AI 写了代码，但生产环境不调用 LLM）：**Layer 3 不存在**，上线后的运维就是传统 SRE。
- 如果产品中包含 LLM 调用（AI 在运行时被调用）：「LLM 返回格式不对」「超时」「挂了」等异常处理**是开发阶段就应该做的**（属于 Layer 2）。即使模型升级导致 breaking change，这和外部的 API dependency 升级没有本质区别——就是普通生产运维，不是新物种。
- 当你在 dev time 为「含 LLM 调用的产品」做开发时，LLM 本身的不确定性在开发阶段就已经被处理了（错误处理、fallback 等）。这些处理属于 Layer 2 的范围。

**因此，不存在一个独立的「Runtime AI Harness Layer」。**

### Layer 4 — 组织治理 Harness

**问题**：「团队里 5 个人各用各的 Agent、各写各的 prompt、各产出各的风格——代码库变成了考古现场。」

**确定性框架**：

| 机制 | 作用 |
|------|------|
| 共享 CLAUDE.md / .cursorrules | 团队级 Agent 行为约束 |
| Prompt 审查 | 关键场景的 prompt 走 review |
| AI 使用审计日志 | 谁让 Agent 改了什么、什么时候 |
| 所有权边界 | Agent 不能改某些关键路径（支付、鉴权） |
| Agent 产出指标 | 追踪 AI 辅助代码的 bug 率 vs 人类代码 |

核心矛盾：**个人效率（自由用 Agent）vs 集体确定性（统一的代码质量）。**

---

### Meta — 持续改进

**问题**：「我怎么知道我的 harness 配置还在起效？模型升级了、代码库长大了、团队变化了——harness 需要演进。」

这是改进 harness 本身的 harness：

| 机制 | 作用 |
|------|------|
| Agent 输出采样审查 | 定期抽查 Agent 产出 |
| Harness 配置版本化 | CLAUDE.md 也在 git 里，能 diff 能 blame |
| A/B 测试约束 | 改了一条规则后，对比产出质量 |
| 事后复盘 | 线上 bug 根因分析 → 是哪层 harness 没兜住？|

这层最容易**被忽略**——因为所有人都在忙着搭前面几层。但它决定了整个 harness 栈不腐烂。

## 递归结构：同一个模式在不同层级的重复

```
产品方 harness
┌──────────────────────────────────┐
│  确定性框架                        │
│  ┌────────────────────────────┐  │
│  │ 非确定性核心 (LLM)          │  │
│  │ · 幻觉 · 风格不一致 · 健忘  │  │
│  └────────────────────────────┘  │
│  输出：可靠的 Agent 工具           │
└──────────────────────────────────┘

用户方 harness
┌──────────────────────────────────┐
│  确定性框架                        │
│  ┌────────────────────────────┐  │
│  │ 非确定性核心 (Agent)         │  │
│  │ · 误判需求 · 代码bug · 过设计│  │
│  └────────────────────────────┘  │
│  输出：可靠的软件产品               │
└──────────────────────────────────┘
```

**区别只在于谁是「核心」，谁是「框架」。** 上一层的输出变成下一层的非确定核心。这种递归嵌套意味着：每一层都在试图用确定性机制消解上一层的残留不确定性。

## 控制力分布：你能控什么

```
              Anthropic 控                   你控
              ─────────────                  ──────
LLM 本身      ✗ (模型能力上限)               ✗
────────
System Prompt ✓ (CLAUDE.md 被注入)          ✓✓ (CLAUDE.md 你写的)
Tool 定义     ✓ (Schema, 权限)              ✗
Workflow      ✓ (pipeline/parallel)         ✓ (你决定任务拆多细)
────────
隔离机制      ✓ (worktree 选项)             ✓✓ (git branch/worktree)
验证机制      ✓ (StructuredOutput)          ✓✓ (你写测试/审查diff)
回滚机制      ✗                              ✓✓ (git reset)
约束注入      ✓ (system prompt)             ✓✓ (CLAUDE.md + rules)
审查机制      ✗                              ✓✓ (你的眼睛 + 第二个Agent)
────────
风格控制      弱 (prompt 里写)               ✓✓✓ (CLAUDE.md + lint + prettier)
正确性验证    弱 (Agent 无法自己验证)         ✓✓✓ (测试套件 + type-check)
```

核心洞察：**越靠近最终产出，你的控制杠杆越多。** 产品方 harness 只能做到「不出明显的错」，而「出你想要的正确结果」只能靠你搭建的外部 harness。

## 关键设计选择总结

| 选择 | 推荐 | 理由 |
|------|------|------|
| Pipeline vs Barrier | 默认 Pipeline | 不必要的 barrier 浪费吞吐量 |
| 测试谁写 | 你写测试，Agent 写实现 | 你知道什么是对的，Agent 知道怎么写 |
| Agent 产出 | 多轮小批量 | 每轮可验证、可回滚 |
| 约束方式 | CLAUDE.md 越具体越好 | 减少 Agent 自由发挥空间 |
| 隔离策略 | 按需 Worktree | 不默认开启，只在冲突风险时用 |
| 审查重点 | 测试逻辑 > 实现代码 | 审查测试比审查实现更高效 |
| 交付形态 | Managed API 已足够 | 除非你自部署推理，否则不需要关心内部分层 |

## 一个好 Agent 任务的属性

| 属性 | 含义 | 验证方式 |
|------|------|----------|
| **原子性** | 改一个关注点 | diff 聚焦在一个主题 |
| **可测试** | 改完能跑测试 | 已有测试或新增测试 |
| **可回滚** | 一个 commit 撤销 | `git revert` 无冲突 |
| **可审查** | diff < 200 行 | 人类能真正 review |

## AI 时代开发者的价值

这个框架也回答了「AI 时代开发者还值钱在哪」——不是写代码（Agent 比你快），而是**设计让 Agent 产出的代码能用的质量保障体系**。这需要：

- 工程判断力（在速度和可靠性之间做取舍）
- 定义什么是「对」（测试、约束、规范）
- 审查 Agent 输出的能力（识别 bug、过度设计、风格不一致）
- 识别和区分不同 harness 层的能力（知道哪里需要确定性约束、哪里不需要）

**这些，Agent 目前做不了。**
